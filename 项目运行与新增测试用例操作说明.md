# 项目运行与新增测试用例操作说明

## 1. 文档目的

本文档用于说明当前 `DwtAutoTest` 项目的：

- 如何安装与运行项目
- 如何执行现有测试用例
- 执行结果输出到哪里
- 如果要新增一个测试案例，应该如何操作
- 新增 Web / API / DB 步骤时分别怎么写

本文档基于当前项目实际实现整理，可直接作为项目使用手册。

---

## 2. 项目简介

当前项目是一个基于 **Node.js + TypeScript + Magnitude** 的自动化测试执行框架，支持以下三类能力：

- **Web 自动化测试**
- **API 接口测试**
- **MySQL 数据库校验**

当前框架通过 YAML 模板定义测试用例，并按以下固定阶段执行：

1. `beforeActions`
2. `mainSteps`
3. `assertions`
4. `afterActions`

支持输出：

- JSON 执行报告
- Markdown 执行报告
- 失败截图 / 手动截图

```bash
node -v
```

在项目根目录执行：


项目 Web 自动化使用 `magnitude-core`，底层依赖浏览器环境。
如果本机缺少 Chromium 等运行环境，需根据实际提示补齐。

### 3.4 .env 配置

项目根目录需要存在 `.env` 文件。
可以先参考 `.env.example` 创建：

```env
WEB_USER_BASE_URL=https://sit.dowalet.com/user/
WEB_ADMIN_BASE_URL=
API_BASE_URL=
LOGIN_USERNAME=
LOGIN_PASSWORD=
DB_HOST=127.0.0.1
DB_PORT=3306
DB_NAME=
DB_USER=
DB_PASSWORD=
```

说明：

- `WEB_USER_BASE_URL`：user 站点地址
- `WEB_ADMIN_BASE_URL`：admin 站点地址
- `API_BASE_URL`：接口测试使用的 API 基础地址
- `LOGIN_USERNAME` / `LOGIN_PASSWORD`：用户侧默认登录账号密码
- `ADMIN_USERNAME` / `ADMIN_PASSWORD`：管理侧默认登录账号密码
- `DB_*`：数据库连接信息

如果使用 Magnitude，需要本地环境里具备对应 LLM 配置，例如你当前已经配置的 API key / relay 信息。

## 8.1 共用 STEP（共享步骤）

当前项目已支持把公共流程抽成共享步骤模板，供多个测试案例复用。

典型场景：

- 多个案例都需要登录
- 多个案例都需要先获取 token
- 多个案例都需要执行同一段初始化或清理逻辑

共享步骤目录建议放在：

- `cases/shared/`

例如：

- `cases/shared/common/web_login.yaml`

场景用例中通过 `use` 引用：

```yaml
mainSteps:
  - use: common/web_login
    with:
      session: user
      username: "${username}"
      password: "${password}"
      site: user
      url: "#/login?redirect=%2Fdashboard"
```

说明：

- `use`：引用共享步骤 ID
- `with`：给共享步骤模板传参数
- `session`：指定浏览器会话名，同一用例中可并存多个会话
- 当前实现是 **phase 内引用**，即：
  - `mainSteps` 中引用，只会展开共享模板中的 `phases.mainSteps`
  - `beforeActions` / `assertions` / `afterActions` 同理
- 执行前会自动展开成普通步骤，再进入现有校验和执行流程

### 8.2 共享步骤模板结构

共享步骤模板示例：

```yaml
sharedId: common/web_login
description: 通用网页登录流程
params:
  session:
    default: default
  username:
    required: true
  password:
    required: true
  site:
    default: user
  url:
    default: "#/login?redirect=%2Fdashboard"

phases:
  mainSteps:
    - id: open_login
      type: web_open
      name: 打开登录页
      params:
        session: "${session}"
        site: "${site}"
        url: "${url}"

    - id: input_username
      type: web_input
      name: 输入用户名
      params:
        target: "用户名输入框"
        value: "${username}"
```

说明：

- `sharedId`：共享步骤唯一标识
- `params`：共享步骤参数声明
- `required: true`：调用时必须在 `with` 中传值
- `default`：调用时不传则使用默认值
- `phases`：定义这个共享模板在哪个 phase 下提供步骤

### 8.4 多账号与双会话用法

当前项目已支持：

- 不同测试用例配置不同账号
- 同一个测试用例中同时保留 `admin` 和 `user` 两个浏览器会话

推荐变量写法：

```yaml
variables:
  adminAccount:
    username: "${ADMIN_USERNAME}"
    password: "${ADMIN_PASSWORD}"
  userAccount:
    username: "${LOGIN_USERNAME}"
    password: "${LOGIN_PASSWORD}"
```

推荐共享登录写法：

```yaml
mainSteps:
  - use: common/web_login
    with:
      session: admin
      site: admin
      username: "${adminAccount.username}"
      password: "${adminAccount.password}"
      url: "#/login"

  - use: common/web_login
    with:
      session: user
      site: user
      username: "${userAccount.username}"
      password: "${userAccount.password}"
      url: "#/login?redirect=%2Fdashboard"
```

后续任意 Web 步骤都可通过 `params.session` 指定在哪个浏览器会话执行：

```yaml
- id: assert_user_01
  type: web_assert_url
  name: 校验 user 会话跳转结果
  params:
    session: user
    site: user
    expected: "#/dashboard"
    timeout: 30000
```

说明：

- `session` 用来区分浏览器会话，例如 `admin` / `user`
- `site` 用来区分站点，例如 `admin` / `user`
- 当前建议一个 `session` 固定绑定一个 `site`
- 当前实现是**双会话并存、步骤顺序执行**，不是并发调度


```yaml
caseId: TC_SAMPLE_001
caseName: 示例用例
description: 示例说明
tags:
  - smoke
priority: P0
mode: hybrid
config:
  retryCount: 1
  timeout: 30000
  screenshotOnFail: true
  stopOnFailure: true
variables:
  username: "${LOGIN_USERNAME}"

beforeActions: []

mainSteps: []

assertions: []

afterActions: []
```

### 字段说明

#### 顶层字段

- `caseId`：用例唯一编号
- `caseName`：用例名称
- `description`：用例说明
- `tags`：标签数组
- `priority`：优先级，如 `P0 / P1`
- `mode`：当前项目支持 `web / api / hybrid`
- `config`：执行配置
- `variables`：变量定义
- `beforeActions`：前置动作
- `mainSteps`：主流程步骤
- `assertions`：断言步骤
- `afterActions`：收尾动作

#### config 字段

- `retryCount`：当前模板字段已保留，但框架尚未实现真实重试逻辑
- `timeout`：当前可以作为用例配置保留，但具体 Web 断言超时仍建议在步骤里显式写 `timeout`
- `screenshotOnFail`：失败时自动截图
- `stopOnFailure`：某一步失败后是否跳过后续主流程

## 10. 新增测试案例的操作步骤

如果你要新增一个测试案例，推荐按下面流程操作。

### 第 1 步：确定用例类型

先确认你的用例属于哪种：

1. **纯 Web 用例**
2. **纯 API 用例**
3. **Web + API 混合用例**
4. **Web + API + DB 混合用例**

如果涉及页面操作，建议模式使用：

```yaml
mode: hybrid
```

### 第 2 步：复制一个最接近的现有模板

例如：

- 做登录类场景：复制 `cases/scenario/login.yaml`
- 做交易类场景：复制 `cases/scenario/login_trade.yaml`

例如新增一个提现场景：

```text
cases/scenario/withdraw.yaml
```

### 第 3 步：修改用例基本信息

至少修改：

```yaml
caseId: TC_WITHDRAW_001
caseName: 用户提现成功
description: 验证用户登录后可以成功发起提现
```

要求：

- `caseId` 唯一
- `caseName` 清晰可读
- `description` 说明测试目标

### 第 4 步：补充 variables

把会变化的测试数据放进 `variables`：

```yaml
variables:
  username: "${LOGIN_USERNAME}"
  password: "${LOGIN_PASSWORD}"
  amount: 200
```

建议：

- 账号密码、环境差异参数优先走 `.env`
- 业务测试数据放 `variables`
- 步骤中尽量通过 `${变量名}` 引用，不直接写死

### 第 5 步：编排执行阶段

推荐遵循下面的职责分工：

#### beforeActions

放前置准备动作，例如：

- 初始化数据库
- 清理旧数据
- 调用登录接口获取 token

#### mainSteps

放主要业务流程，例如：

- 打开页面
- 输入表单
- 点击按钮
- 调用业务接口

#### assertions

放断言，例如：

- 页面 URL 校验
- 页面文本校验
- 接口字段校验
- 数据库结果校验

#### afterActions

放收尾动作，例如：

- 清理测试数据
- 回收脏数据

注意：

- 即使主流程失败，`afterActions` 仍会尽量执行

## 12. 新增 Web 用例时，定位器如何配置

Web 步骤里的 `target` 不是直接写 CSS，而是写业务名称，例如：

```yaml
target: "登录按钮"
```

真正的 CSS / text 定位器定义在：

- `config/locators/login.yaml`
- `config/locators/trade.yaml`

例如：

```yaml
submit:
  strategy: css
  value: 'button[type="submit"]:visible'

aliases:
  登录按钮: submit
```

### 新增页面元素的操作方法

假设你新增了一个“提现金额输入框”，操作方式如下：

#### 第一步：在 locator 文件中新增定位器

```yaml
withdrawAmount:
  strategy: css
  value: '[data-testid="withdraw-amount"]'
```

#### 第二步：增加中文别名

```yaml
aliases:
  提现金额输入框: withdrawAmount
```

#### 第三步：在 YAML 用例中引用

```yaml
- id: step_05
  type: web_input
  name: 输入提现金额
  params:
    target: "提现金额输入框"
    value: "${amount}"
```

建议：

- 优先使用稳定定位器：`data-testid`、`id`、`name`
- 尽量避免脆弱的层级 CSS
- 如果页面元素属于新业务域，可新增独立 locator 文件并加入 `config/web.yaml`

## 13. 新增 API 测试步骤怎么写

### 13.1 调用接口

```yaml
- id: api_01
  type: api_call
  name: 调用登录接口
  params:
    method: POST
    url: "/api/login"
    headers:
      Content-Type: application/json
    body:
      username: "${username}"
      password: "${password}"
```

说明：

- 相对路径会自动拼接 `config/api.yaml` 中的 `baseUrl`
- 支持 `headers`
- 支持 `query`
- 支持 `body`

### 13.2 提取接口返回值到上下文

```yaml
extract:
  token: "$.data.token"
  userId: "$.data.userId"
```

提取后，后续步骤可以使用：

```yaml
Authorization: "Bearer ${context.token}"
```

### 13.3 接口字段断言

```yaml
- id: assert_api_01
  type: api_assert_field
  name: 校验返回码
  params:
    field: "$.code"
    expected: 0
```

注意：

- `api_assert_field` 默认断言最近一次 `api_call` 的响应 body

## 14. 新增数据库测试步骤怎么写

### 14.1 初始化数据

```yaml
- id: before_db_01
  type: db_init
  name: 初始化账户余额
  params:
    sql: "UPDATE user_account SET balance = 100000 WHERE user_id = 10001"
```

### 14.2 清理数据

```yaml
- id: after_db_01
  type: db_cleanup
  name: 清理测试订单
  params:
    sql: "DELETE FROM trade_order WHERE trade_id = '${context.tradeId}'"
```

### 14.3 数据库断言

```yaml
- id: assert_db_01
  type: db_assert
  name: 校验订单存在
  params:
    sql: "SELECT count(*) AS cnt FROM trade_order WHERE trade_id = '${context.tradeId}'"
    expected:
      cnt: 1
```

说明：

- `db_assert` 会取查询结果第一行进行比对
- `expected` 中写要校验的字段和值

### 14.4 SQL 安全限制

数据库步骤会做 SQL 安全校验。

请避免：

- `drop`
- `truncate`
- 无 `where` 的 `delete`
- 无 `where` 的 `update`

否则模板校验阶段就会失败。

## 16. 新增一个完整测试用例的推荐流程

推荐按以下顺序新增：

### 方案步骤

1. 在 `cases/scenario/` 新建 YAML 文件
2. 写好 `caseId`、`caseName`、`description`
3. 明确 `variables`
4. 先写 `beforeActions`
5. 再写 `mainSteps`
6. 最后写 `assertions`
7. 如果需要清理数据，再写 `afterActions`
8. 如果有新的页面元素，更新 `config/locators/*.yaml`
9. 执行 `npm run typecheck`
10. 执行 `npm start -- <你的yaml路径>`
11. 查看 `reports/` 和 `screenshots/`

## 17. 新增用例示例

下面给一个简化示例：

```yaml
caseId: TC_PROFILE_001
caseName: 用户进入个人中心
description: 验证用户登录后可以进入个人中心页面
tags:
  - smoke
  - web
priority: P1
mode: hybrid

  retryCount: 1
  timeout: 30000
  screenshotOnFail: true
  stopOnFailure: true
variables:
  username: "${LOGIN_USERNAME}"
  password: "${LOGIN_PASSWORD}"

beforeActions: []

mainSteps:
  - id: step_01
    type: web_open
    name: 打开登录页
    params:
      site: user
      url: "#/login?redirect=%2Fdashboard"

  - id: step_02
    type: web_input
    name: 输入用户名
    params:
      target: "用户名输入框"
      value: "${username}"

  - id: step_03
    type: web_input
    name: 输入密码
    params:
      target: "密码输入框"
      value: "${password}"

  - id: step_04
    type: web_click
    name: 点击登录按钮
    params:
      target: "登录按钮"

  - id: step_05
    type: web_click
    name: 点击个人中心菜单
    params:
      target: "个人中心菜单"

assertions:
  - id: assert_01
    type: web_assert_url
    name: 校验进入个人中心
    params:
      site: user
      expected: "#/profile"
      timeout: 30000

afterActions: []
```

如果这里新增了 `个人中心菜单`，记得同步补 locator 配置。

## 18. 常见问题

### 18.1 为什么 target 不能直接写 CSS？

可以写业务别名是为了让测试脚本更可读，页面元素变化时也只需要维护 locator 文件，不必到处改 YAML。

### 18.2 为什么我的 `${context.xxx}` 报缺少来源？

因为你引用了上下文变量，但前面的步骤没有通过 `extract` 产出它。

例如用了：

```yaml
${context.token}
```

那前面必须有类似：

```yaml
extract:
  token: "$.data.token"
```

### 18.3 为什么数据库步骤执行失败？

常见原因：

- `.env` 中数据库配置不完整
- SQL 不安全，被拦截
- 断言字段与查询结果不一致

### 18.4 为什么执行通过了但断言失败？

说明步骤能执行，但业务结果与预期不一致。优先检查：

- 页面是否真实跳转
- 文本是否真实出现
- API 返回值是否符合预期
- 数据库查询条件是否正确

## 19. 推荐实践

建议团队新增用例时统一遵循：

1. 一个 YAML 文件只描述一个清晰场景
2. `caseId` 必须唯一
3. 优先把账号、密码、地址放 `.env`
4. 优先使用稳定 locator
5. 所有关键业务结果都要有断言
6. 涉及脏数据的场景必须写 `afterActions`
7. 新增后先跑一遍单用例，再交付

## 21. 一句话总结

如果你要新增测试案例，最核心的操作就是：

**复制一个现有 YAML 模板 -> 修改业务步骤 -> 补齐变量 -> 配置 locator -> 运行命令 -> 查看报告。**
