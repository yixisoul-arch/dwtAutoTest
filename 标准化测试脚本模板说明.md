# 标准化测试脚本模板说明

# 1. 文档说明

| 项目 | 内容 |
|---|---|
| 文档名称 | 标准化测试脚本模板说明 |
| 版本 | V1.0 |
| 适用项目 | 基于 Magnitude 的 AI 自动化测试平台 |
| 目的 | 统一定义测试脚本模板结构、字段规范、步骤类型、变量规则、断言规则与示例，保证测试用例可生成、可执行、可维护 |

---

# 2. 设计目标

标准化测试脚本模板用于承接自然语言生成结果，并作为系统最终可执行的测试资产。

设计目标如下：

- 统一测试用例描述格式
- 支持 Web、API、数据库三类能力组合编排
- 支持前置初始化、执行、断言、后置清理完整闭环
- 支持上下文变量传递
- 支持 AI 生成后人工审查与维护
- 支持后续扩展更多步骤类型和断言类型

---

# 3. 模板设计原则

## 3.1 可执行
模板字段必须足够明确，能够直接转换为执行计划。

## 3.2 可读
模板结构需便于测试、开发、产品共同理解。

## 3.3 可维护
步骤字段命名统一，支持复用、参数化和拆分。

## 3.4 可扩展
允许后续新增步骤类型、断言类型、上下文能力。

## 3.5 安全可控
数据库、接口、页面操作均需具备明确边界，避免不受控执行。

---

# 4. 推荐模板格式

首期建议采用 **YAML** 作为标准化测试脚本模板格式。

原因如下：

- 可读性高
- 结构清晰
- 适合人工编写与 AI 生成
- 支持嵌套结构表达复杂步骤
- 便于后续转为 JSON 对象执行

---

# 5. 模板总体结构

一个完整测试用例建议由以下部分组成：

```yaml
caseId: string
caseName: string
description: string
tags: []
priority: P0 | P1 | P2
mode: web | api | hybrid
config: {}
variables: {}
beforeActions: []
mainSteps: []
assertions: []
afterActions: []
```

字段说明如下：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| caseId | string | 是 | 用例唯一标识 |
| caseName | string | 是 | 用例名称 |
| description | string | 是 | 用例说明 |
| tags | array | 否 | 标签，如 login、trade、smoke |
| priority | string | 否 | 优先级，建议 P0/P1/P2 |
| mode | string | 是 | 执行模式：web / api / hybrid |
| config | object | 否 | 用例级配置 |
| variables | object | 否 | 入参或初始变量 |
| beforeActions | array | 否 | 前置初始化步骤 |
| mainSteps | array | 是 | 主执行步骤 |
| assertions | array | 否 | 独立断言步骤 |
| afterActions | array | 否 | 后置清理步骤 |

# 6. 顶层字段规范

## 6.1 caseId
建议命名规则：

```text
TC_模块_场景_序号
```

示例：

```text
TC_LOGIN_001
TC_TRADE_CREATE_001
TC_LOGIN_TRADE_001
```

## 6.2 caseName
使用可读性高的中文或业务名称。

示例：

```text
用户登录成功
登录后发起交易并校验数据库记录
```

## 6.3 description
说明测试目标，不写实现细节。

示例：

```text
验证用户成功登录后发起交易，系统页面提示成功，接口返回正确，数据库生成交易记录。
```

## 6.4 tags
推荐用于：

- 场景分类：login、trade
- 测试类型：smoke、regression
- 能力分类：web、api、db

## 6.5 mode
可选值：

- `web`：仅页面测试
- `api`：仅接口测试
- `hybrid`：页面 + API + 数据库组合测试

# 7. config 配置规范

`config` 用于定义当前用例特有的执行策略。

示例：

```yaml
config:
  retryCount: 1
  timeout: 10000
  screenshotOnFail: true
  stopOnFailure: true
```

字段建议：

| 字段 | 类型 | 说明 |
|---|---|---|
| retryCount | number | 失败重试次数 |
| timeout | number | 默认超时，单位毫秒 |
| screenshotOnFail | boolean | 失败是否截图 |
| stopOnFailure | boolean | 单步失败后是否中止 |

---

# 8. variables 变量规范

`variables` 用于定义测试执行时的初始变量。

示例：

```yaml
variables:
  username: test_user
  password: test_pass
  tradeAmount: 100
  currency: CNY
```

## 8.1 变量引用规则
模板中变量使用 `${变量名}` 形式引用。

示例：

```yaml
value: "${username}"
```

## 8.2 上下文变量引用
系统执行过程中产生的动态数据也通过 `${}` 引用，例如：

```yaml
value: "${context.tradeId}"
```

推荐上下文命名：

- `${context.token}`
- `${context.userId}`
- `${context.tradeId}`
- `${context.lastResponse.code}`

# 9. 步骤结构规范

除普通步骤外，当前版本还支持“共享步骤引用”。共享步骤用于把登录、获取 token、清理测试数据等公共流程抽到独立模板中复用。

## 9.1 普通步骤结构

所有普通步骤统一采用以下基础结构：

```yaml
- id: step_01
  type: step_type
  name: 步骤名称
  enabled: true
  params: {}
  extract: {}
```

字段说明：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| id | string | 是 | 步骤唯一标识 |
| type | string | 是 | 步骤类型 |
| name | string | 是 | 步骤名称 |
| enabled | boolean | 否 | 是否启用，默认 true |
| params | object | 是 | 当前步骤参数 |
| extract | object | 否 | 从执行结果中提取变量 |

## 9.2 共享步骤引用结构

在 `beforeActions` / `mainSteps` / `assertions` / `afterActions` 中，也可以写共享步骤引用项：

```yaml
- use: common/web_login
  with:
    session: user
    username: "${username}"
    password: "${password}"
    site: user
    url: "#/login?redirect=%2Fdashboard"
```

字段说明：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| use | string | 是 | 共享步骤 ID，对应 `cases/shared/` 下的模板 |
| with | object | 否 | 共享步骤参数传值 |
| enabled | boolean | 否 | 是否启用该共享引用 |
| idPrefix | string | 否 | 可选的展开后步骤 ID 前缀 |

补充说明：

- `with.session` 可用于指定浏览器会话名，例如 `admin` / `user`
- 一个用例中可多次引用同一共享步骤，并传入不同的 `session/site/username/password`
- 当前推荐一个 `session` 固定绑定一个 `site`

说明：

- 当前版本采用 **phase 内引用** 模式
- `mainSteps` 中引用，只展开共享模板里的 `phases.mainSteps`
- `beforeActions` / `assertions` / `afterActions` 同理
- 展开后的步骤会自动改写 `id`，避免与其他步骤冲突

## 9.3 共享步骤模板结构

共享步骤模板建议放在 `cases/shared/`，示例：

```yaml
sharedId: common/web_login
description: 通用网页登录流程
params:
  session:
    default: default
  username:
    required: true
  password:
    required: true
  site:
    default: user
  url:
    default: "#/login?redirect=%2Fdashboard"

phases:
  mainSteps:
    - id: open_login
      type: web_open
      name: 打开登录页
      params:
        session: "${session}"
        site: "${site}"
        url: "${url}"
```

字段说明：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| sharedId | string | 是 | 共享步骤唯一标识 |
| description | string | 否 | 共享步骤说明 |
| params | object | 否 | 参数声明与默认值 |
| phases | object | 是 | 按 phase 提供可展开步骤 |

`params` 中支持：

- `required: true`：调用时必须传值
- `default`：调用时不传则使用默认值


# 10. beforeActions 规范

`beforeActions` 用于执行测试前准备动作，通常包括：

- 清理历史测试数据
- 插入测试依赖数据
- 调用接口准备前置数据
- 获取登录态或初始 token

示例：

```yaml
beforeActions:
  - id: before_01
    type: db_cleanup
    name: 清理旧交易数据
    params:
      sql: "DELETE FROM trade_order WHERE test_flag = 'Y'"

  - id: before_02
    type: db_init
    name: 初始化账户数据
    params:
      sql: "INSERT INTO user_account(id, balance, test_flag) VALUES (10001, 100000, 'Y')"
```

# 11. mainSteps 规范

`mainSteps` 是主执行过程，通常包括页面、接口、数据处理等步骤。

---

# 12. assertions 规范

`assertions` 用于定义独立断言步骤。

适用场景：

- 页面断言与执行动作分离
- 接口断言统一管理
- 数据库断言统一管理

示例：

```yaml
assertions:
  - id: assert_01
    type: web_assert_text
    name: 校验首页欢迎语
    params:
      target: "欢迎区域"
      expected: "欢迎登录"

  - id: assert_02
    type: db_assert
    name: 校验交易记录已生成
    params:
      sql: "SELECT count(*) AS cnt FROM trade_order WHERE trade_no = '${context.tradeNo}'"
      expected:
        cnt: 1
```

---

# 13. afterActions 规范

`afterActions` 用于测试后清理动作，避免污染环境。

示例：

```yaml
afterActions:
  - id: after_01
    type: db_cleanup
    name: 清理交易测试数据
    params:
      sql: "DELETE FROM trade_order WHERE test_flag = 'Y'"
```

# 14. 支持的步骤类型定义

首期建议支持以下步骤类型。

## 14.1 Web 类步骤

### 14.1.1 web_open
打开页面。

```yaml
- id: step_01
  type: web_open
  name: 打开登录页
  params:
    url: "/login"
```

参数说明：

| 参数 | 类型 | 必填 | 说明 |
|---|---|---|---|
| url | string | 是 | 页面地址，可相对 baseUrl |

---

### 14.1.2 web_input
输入内容。

```yaml
- id: step_02
  type: web_input
  name: 输入用户名
  params:
    target: "用户名输入框"
    value: "${username}"
```

参数说明：

| 参数 | 类型 | 必填 | 说明 |
|---|---|---|---|
| target | string | 是 | 目标元素描述或定位标识 |
| value | string | 是 | 输入值 |

---

### 14.1.3 web_click
点击元素。

```yaml
- id: step_03
  type: web_click
  name: 点击登录按钮
  params:
    target: "登录按钮"
```

---

### 14.1.4 web_select
下拉选择。

```yaml
- id: step_04
  type: web_select
  name: 选择币种
  params:
    target: "币种下拉框"
    value: "CNY"
```

---

### 14.1.5 web_wait_for_element
等待元素出现。

```yaml
- id: step_05
  type: web_wait_for_element
  name: 等待交易表单出现
  params:
    target: "交易表单"
    timeout: 5000
```

---

### 14.1.6 web_assert_text
断言文本。

```yaml
- id: step_06
  type: web_assert_text
  name: 校验提示文案
  params:
    target: "页面提示"
    expected: "提交成功"
```

---

### 14.1.7 web_assert_visible
断言元素可见。

```yaml
- id: step_07
  type: web_assert_visible
  name: 校验欢迎模块可见
  params:
    target: "欢迎模块"
```

---

### 14.1.8 web_assert_url
断言页面地址。

```yaml
- id: step_08
  type: web_assert_url
  name: 校验跳转首页
  params:
    expected: "/home"
```

---

### 14.1.9 web_screenshot
页面截图。

```yaml
- id: step_09
  type: web_screenshot
  name: 保存成功页面截图
  params:
    fileName: "trade_success.png"
```

---

## 14.2 API 类步骤

### 14.2.1 api_call
执行接口请求。

```yaml
- id: step_10
  type: api_call
  name: 调用创建交易接口
  params:
    method: POST
    url: "/api/trade/create"
    headers:
      Authorization: "Bearer ${context.token}"
    body:
      amount: "${tradeAmount}"
      currency: "${currency}"
  extract:
    tradeId: "$.data.tradeId"
    code: "$.code"
```

参数说明：

| 参数 | 类型 | 必填 | 说明 |
|---|---|---|---|
| method | string | 是 | 请求方法 |
| url | string | 是 | 请求地址 |
| headers | object | 否 | 请求头 |
| query | object | 否 | 查询参数 |
| body | object | 否 | 请求体 |
| timeout | number | 否 | 请求超时 |

`extract` 用于从响应中提取上下文变量。

---

### 14.2.2 api_assert_status
断言状态码。

```yaml
- id: step_11
  type: api_assert_status
  name: 校验接口状态码
  params:
    expected: 200
```

---

### 14.2.3 api_assert_field
断言响应字段。

```yaml
- id: step_12
  type: api_assert_field
  name: 校验业务返回码
  params:
    field: "$.code"
    expected: 0
```

---

### 14.2.4 api_assert_contains
断言响应包含内容。

```yaml
- id: step_13
  type: api_assert_contains
  name: 校验响应包含成功信息
  params:
    field: "$.message"
    expected: "success"
```

---

## 14.3 数据库类步骤

### 14.3.1 db_init
执行初始化 SQL。

```yaml
- id: step_14
  type: db_init
  name: 初始化账户余额
  params:
    sql: "UPDATE user_account SET balance = 100000 WHERE user_id = 10001"
```

---

### 14.3.2 db_query
执行查询 SQL。

```yaml
- id: step_15
  type: db_query
  name: 查询交易记录
  params:
    sql: "SELECT trade_no, amount, status FROM trade_order WHERE trade_id = '${context.tradeId}'"
  extract:
    dbTradeNo: "[0].trade_no"
    dbAmount: "[0].amount"
```

---

### 14.3.3 db_assert
执行数据库断言。

```yaml
- id: step_16
  type: db_assert
  name: 校验交易记录存在
  params:
    sql: "SELECT count(*) AS cnt FROM trade_order WHERE trade_id = '${context.tradeId}'"
    expected:
      cnt: 1
```

说明：
- `expected` 中定义预期字段和值
- 执行器负责比较查询结果与预期是否一致

---

### 14.3.4 db_cleanup
执行清理 SQL。

```yaml
- id: step_17
  type: db_cleanup
  name: 清理测试交易数据
  params:
    sql: "DELETE FROM trade_order WHERE trade_id = '${context.tradeId}'"
```

# 15. extract 规则

`extract` 用于从步骤结果中提取变量写入上下文。

## 15.1 API 提取示例

```yaml
extract:
  token: "$.data.token"
  userId: "$.data.userId"
```

## 15.2 数据库提取示例

```yaml
extract:
  orderId: "[0].order_id"
```

规则说明：

- API 提取建议使用 JSONPath 风格
- 数据库查询结果建议使用数组下标路径
- 提取结果统一写入 `context`

引用方式：

```yaml
"${context.token}"
"${context.orderId}"
```

# 16. 断言规则

## 16.1 页面断言
建议支持：

- 文本相等
- 文本包含
- 元素可见
- 元素可点击
- URL 匹配

## 16.2 API 断言
建议支持：

- HTTP 状态码
- 返回字段值
- 返回内容包含关系
- 响应时间阈值

## 16.3 数据库断言
建议支持：

- 查询结果条数
- 字段值匹配
- 状态变化匹配
- 数值范围匹配

# 17. 变量替换规则

## 17.1 替换顺序
建议变量替换顺序如下：

1. 当前用例 `variables`
2. 运行时 `context`
3. 系统默认配置变量

## 17.2 示例

```yaml
body:
  userId: "${context.userId}"
  amount: "${tradeAmount}"
```

# 19. 安全约束

为保证模板执行安全，需遵守以下规则：

## 19.1 数据库安全
- 仅允许测试环境数据库
- 禁止 `drop`、`truncate`
- 禁止无条件 `delete`、`update`
- 建议所有测试数据带 `test_flag`

## 19.2 接口安全
- 禁止调用生产环境接口
- token、密码等信息不写入报告明文

## 19.3 页面安全
- 禁止执行具有破坏性的生产操作
- 页面关键步骤建议增加确认断言

# 20. 完整示例一：登录场景

```yaml
caseId: TC_LOGIN_001
caseName: 用户登录成功
description: 验证用户输入正确账号密码后可以成功登录系统

      url: "/login"

  - id: step_02
    type: web_input
    name: 输入用户名

      target: "用户名输入框"

  - id: step_03
    type: web_input
    name: 输入密码

  - id: step_04
    type: web_click
    name: 点击登录按钮
    params:
      target: "登录按钮"

assertions:
  - id: assert_01
    type: web_assert_url
    name: 校验跳转首页
    params:
      expected: "/home"

  - id: assert_02
    type: web_assert_text
    name: 校验欢迎语存在

```

# 22. AI 生成模板约束建议

为提高 AI 生成结果质量，建议约束如下：

1. AI 必须输出标准 YAML 结构
2. 每个步骤必须包含 `id`、`type`、`name`、`params`
3. 所有变量必须通过 `${}` 引用
4. 所有数据库 SQL 必须明确 where 条件
5. 所有关键业务步骤必须有至少一个断言
6. 涉及数据库写入的测试必须有 `afterActions`
