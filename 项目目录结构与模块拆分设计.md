# 项目目录结构与模块拆分设计

# 1. 文档说明

| 项目 | 内容 |
|---|---|
| 文档名称 | 项目目录结构与模块拆分设计 |
| 版本 | V1.0 |
| 适用项目 | 基于 Magnitude 的 AI 自动化测试平台 |
| 目的 | 明确项目落地时的目录组织方式、模块职责边界、模块依赖关系与建议实施顺序 |

---

# 2. 设计目标

目录结构与模块拆分设计的目标如下：

- 保证项目结构清晰，便于多人协作
- 保证 Web、API、数据库、AI、报告等能力边界明确
- 保证测试模板、配置、执行引擎、公共能力相互解耦
- 为后续扩展更多场景、更多模板和更多能力预留空间

# 4. 顶层目录说明

## 4.1 docs/
存放项目级文档，包括需求、技术方案、模板说明、开发计划等。

### 建议内容
- PRD
- 技术方案
- 模板规范
- 开发任务拆解
- 模块设计说明

---

## 4.2 config/
存放系统配置文件。

### 建议拆分
- `system.yaml`：系统通用配置
- `web.yaml`：Web 执行配置
- `api.yaml`：API 执行配置
- `db.yaml`：数据库连接配置
- `report.yaml`：报告输出配置

### 设计要求
- 支持统一加载
- 支持默认值
- 支持环境变量覆盖
- 敏感信息不直接硬编码

---

## 4.3 cases/
存放标准化测试资产。

### 子目录建议
- `web/`：纯页面测试用例
- `api/`：纯接口测试用例
- `scenario/`：场景级混合测试用例
- `templates/`：模板片段、通用模板、基线模板

### 适用内容
- 登录场景模板
- 发起交易场景模板
- 公共模板片段
- 示例模板

---

## 4.4 src/

# 5. src 目录模块拆分设计

# 5.1 src/ai/
AI 能力模块，负责自然语言理解和辅助生成能力。

## 5.1.1 parser/
### 职责
- 解析自然语言需求
- 提取页面动作、接口动作、数据库动作
- 提取场景目标和断言目标

### 输出
- 结构化意图对象

---

## 5.1.2 generator/
### 职责
- 根据结构化意图生成 YAML 模板草稿
- 补齐步骤 ID、默认断言、默认结构

### 输出
- 标准化测试脚本模板

---

## 5.1.3 recommender/
### 职责
- 推荐断言点
- 推荐数据库校验点
- 推荐 API 校验点
- 推荐页面步骤补全建议

---

## 5.1.4 analyzer/
### 职责
- 分析失败结果
- 输出失败类型与可能原因
- 输出建议检查点

---

# 5.2 src/engine/
执行引擎模块，负责将步骤真正执行起来。

## 5.2.1 web/
### 职责
- 接入 Magnitude 页面能力
- 实现页面打开、输入、点击、断言、截图等

### 建议文件
```text
src/engine/web/
├─ web_engine.*
├─ web_actions.*
├─ web_assertions.*
└─ locator_strategy.*
```

---

## 5.2.2 api/
### 职责
- 实现接口调用
- 实现响应提取
- 实现状态码与字段断言

### 建议文件
```text
src/engine/api/
├─ api_engine.*
├─ api_client.*
└─ api_assertions.*
```

---

## 5.2.3 db/
### 职责
- 连接 MySQL 8.0
- 执行初始化、查询、断言、清理
- 拦截危险 SQL

### 建议文件
```text
src/engine/db/
├─ db_engine.*
├─ db_client.*
├─ db_assertions.*
└─ sql_guard.*
```

---

## 5.2.4 orchestration/
### 职责
- 统一编排 before/main/assertion/after 步骤
- 根据步骤类型路由到 Web / API / DB 引擎
- 控制 stopOnFailure、重试、超时等策略

### 建议文件
```text
src/engine/orchestration/
├─ plan_builder.*
├─ dispatcher.*
├─ executor.*
└─ step_router.*
```

# 6. 模块依赖关系建议

建议整体依赖方向如下：

```text
main
  -> parser / validator / engine / report / context / model / common
engine/orchestration
  -> engine/web / engine/api / engine/db / context / model / common
engine/web
  -> common / model
engine/api
  -> common / model / context
engine/db
  -> common / model / context
ai/generator
  -> ai/parser / model / common
report
  -> model / common
parser
  -> model / common
validator
  -> parser / model / common
```

## 设计约束
- `common` 不依赖业务模块
- `model` 尽量纯净，不依赖执行模块
- `engine/web`、`engine/api`、`engine/db` 之间不直接耦合
- 跨能力联动通过 `context` 和 `orchestration` 实现

# 7. 推荐落地顺序

建议模块实现顺序如下：

## 第一步：基础层
- `common`
- `model`
- `config`
- `parser`

## 第二步：执行底座
- `context`
- `engine/db`
- `engine/api`
- `engine/web`
- `engine/orchestration`

## 第三步：输出层
- `report`
- `logs`
- `screenshots`

## 第四步：AI 能力层
- `ai/parser`
- `ai/generator`
- `ai/recommender`
- `ai/analyzer`

## 第五步：校验与增强
- `validator`
- 稳定性增强
- 安全控制增强

- 登录页 UI 校验

## 8.2 api/

- 登录接口

适合存放模板片段，例如：

# 9. 文件命名建议

## 9.1 文档命名
统一中文或统一英文均可，但建议当前项目保持中文文档命名一致。

## 9.2 用例模板命名
建议使用业务语义 + 类型，例如：

```text
login_success.yaml
trade_create_success.yaml
login_trade_e2e.yaml
```

若保持中文，也建议简洁：

```text
登录场景测试模板.yaml
发起交易场景测试模板.yaml
```

## 9.3 代码文件命名
建议统一小写下划线或小写短横线风格，避免混用。

# 11. 结论

本目录结构与模块拆分方案的核心思想是：

- 文档、配置、模板、代码、结果分层存放
- Web / API / DB 能力独立实现
- 场景编排通过 orchestration 统一调度
- AI 能力通过 parser / generator / analyzer 插件式接入
- 报告、日志、截图形成完整可追踪闭环

按该结构落地后，项目将具备较好的可维护性、可扩展性与多人协作基础。
