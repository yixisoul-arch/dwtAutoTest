# 2. 建设目标

基于 PRD 的要求，本项目需要建设一套面向 Web 场景的 AI 自动化测试平台，支持以下核心能力：

- Web 页面自动化测试
- API 自动化测试
- MySQL 8.0 数据初始化、结果校验和测试后清理
- 基于自然语言生成测试步骤与标准化脚本模板
- 输出结构化测试执行报告
- 首期重点支持登录、发起交易场景

本技术方案重点解决以下问题：

1. 如何统一编排页面、API、数据库三类测试能力  
2. 如何将 AI 能力接入测试用例生成与执行过程  
3. 如何设计标准化脚本模板以保证可执行与可维护  
4. 如何形成稳定、可追踪、可扩展的执行与报告链路  

---

# 3. 设计原则

## 3.1 分层解耦
页面测试、接口测试、数据库操作、AI 生成、报告输出解耦设计，避免逻辑耦合过重。

## 3.2 场景驱动
以业务场景为核心组织测试执行链路，支持单场景端到端执行。

## 3.3 模板标准化
自然语言输入最终需落到标准化脚本模板，确保测试资产可维护、可复用。

## 3.4 安全可控
数据库操作仅允许测试环境，严格限制风险操作，保证数据安全。

## 3.5 易扩展
为后续扩展多数据库、多浏览器、多环境、CI/CD 集成保留接口。

# 4. 总体架构设计

## 4.1 总体架构说明

系统总体划分为 6 个核心层次：

1. **输入层**
   - 自然语言测试需求输入
   - 标准化测试脚本输入

2. **编排层**
   - 测试场景解析
   - 步骤编排
   - 执行计划生成

3. **执行层**
   - Web 自动化执行引擎
   - API 自动化执行引擎
   - 数据库执行引擎

4. **AI 能力层**
   - 自然语言解析
   - 步骤生成
   - 断言推荐

5. **结果处理层**
   - 执行日志采集
   - 截图采集
   - 数据库结果汇总
   - 报告生成

6. **配置与基础设施层**
   - 测试配置管理
   - 数据库连接管理
   - 用例模板管理
   - 日志管理

# 5. 核心模块设计

# 5.1 输入解析模块

## 5.1.1 目标
负责接收用户输入，并统一转换为系统内部可识别的测试任务对象。

## 5.1.2 输入类型
支持两类输入：

1. **自然语言输入**
   - 例如：登录后发起一笔交易，并验证数据库交易记录生成成功

2. **标准化脚本模板输入**
   - 已沉淀的可执行测试用例模板

## 5.1.3 输出结果
输出统一测试任务模型，例如：

```json
{
  "caseName": "登录并发起交易",
  "mode": "hybrid",
  "steps": [],
  "assertions": [],
  "dbActions": []
}
```

---

# 5.2 AI 能力模块

## 5.2.1 模块职责
将自然语言需求转化为结构化测试步骤，并辅助推荐断言点与校验逻辑。

## 5.2.2 子能力设计

### 1）自然语言解析器
负责抽取：

- 测试目标
- 页面动作
- API 调用动作
- 数据库校验点
- 前置初始化与后置清理要求

### 2）步骤生成器
将解析结果转为结构化步骤，例如：

- 打开页面
- 输入账号密码
- 点击登录
- 调用交易接口
- 执行 SQL 校验交易记录

### 3）断言推荐器
针对页面/API/数据库自动补充常见断言点，例如：

- 页面跳转成功
- 接口状态码为 200
- 数据库新增一条交易记录

### 4）失败分析辅助器
结合执行日志、报错信息、截图，输出初步失败原因分类。

## 5.2.3 输出约束
AI 输出结果不直接作为最终执行内容，必须转换为标准化脚本模板后再执行，确保结果稳定、可审查、可维护。

---

# 5.3 场景编排模块

## 5.3.1 模块职责
负责将结构化测试步骤组装成完整的执行计划。

## 5.3.2 编排内容
编排层需处理以下内容：

- 前置数据初始化
- 页面步骤执行顺序
- API 步骤执行顺序
- 页面/API 结果依赖关系
- 数据库校验步骤插入
- 测试后清理步骤插入

## 5.3.3 执行计划模型
建议内部统一执行计划结构：

```json
{
  "caseId": "TC_LOGIN_TRADE_001",
  "caseName": "登录并发起交易",
  "beforeActions": [],
  "mainSteps": [],
  "assertions": [],
  "afterActions": []
}
```

---

# 5.4 Web 自动化执行模块

## 5.4.1 模块职责
基于 Magnitude 提供 Web 页面访问、元素识别、交互操作和页面断言能力。

## 5.4.2 能力范围
支持以下操作类型：

- openPage
- input
- click
- select
- upload
- hover
- scroll
- waitForElement
- waitForText
- assertText
- assertVisible
- assertUrl
- screenshot

## 5.4.3 元素定位策略
建议定位策略优先级如下：

1. 明确唯一标识（id / data-testid）
2. 稳定属性定位
3. 文本语义定位
4. 相对路径定位

不建议强依赖脆弱 XPath 作为主定位方案。

## 5.4.4 稳定性设计
为降低页面波动带来的失败风险，需支持：

- 显式等待
- 元素重试查找
- 页面加载状态检测
- 关键步骤截图

---

# 5.5 API 自动化执行模块

## 5.5.1 模块职责
负责执行标准 HTTP 请求，并对返回结果进行断言。

## 5.5.2 核心能力
支持：

- 请求构建
- Header 注入
- Query 参数注入
- Body 注入
- Token/鉴权信息注入
- 返回结果提取
- 响应断言

## 5.5.3 返回结果处理
支持将 API 返回值写入上下文，供后续步骤使用，例如：

- 提取 token 用于登录态
- 提取 tradeId 用于数据库校验
- 提取业务主键用于页面查询

---

# 5.6 数据库执行模块

## 5.6.1 模块职责
负责执行测试前初始化、测试中校验、测试后清理。

## 5.6.2 能力划分

### 1）初始化能力
- 删除旧测试数据
- 插入前置依赖数据
- 更新初始状态

### 2）校验能力
- 查询业务记录
- 核对字段值
- 核对状态流转
- 核对数量变化

### 3）清理能力
- 删除新增数据
- 恢复变更数据
- 清理本次执行污染数据

## 5.6.3 安全控制
数据库模块必须满足：

- 固定连接测试环境 MySQL 8.0
- 执行白名单控制
- 危险 SQL 拦截
- 敏感字段脱敏输出

建议限制如下高风险 SQL：

- drop
- truncate
- 无 where 的 delete / update

---

# 5.7 上下文管理模块

## 5.7.1 模块职责
负责在整个测试执行过程中共享数据，保证步骤间可以传递上下文变量。

## 5.7.2 上下文内容示例
包括但不限于：

- 当前登录 token
- 用户 ID
- tradeId
- 页面输入参数
- API 返回结果
- 数据库查询结果
- 执行时间戳

## 5.7.3 典型使用方式
例如：

- 页面提交交易后，从接口或页面结果提取 tradeId
- 后续数据库校验步骤使用该 tradeId 查询记录

---

# 5.8 报告生成模块

## 5.8.1 模块职责
负责汇总执行结果并生成结构化测试报告。

## 5.8.2 报告内容
报告应包含：

- 测试任务信息
- 用例信息
- 执行步骤列表
- 成功/失败状态
- 错误信息
- 页面截图路径
- API 请求与响应摘要
- 数据库执行与断言摘要

## 5.8.3 报告格式
建议支持：

- Markdown
- HTML
- JSON

其中：
- Markdown 便于查看和沉淀
- HTML 便于展示
- JSON 便于系统集成

# 6. 标准化测试脚本模板设计

# 6.1 设计目标
将自然语言和测试逻辑沉淀为统一格式，保证：

- 可执行
- 可维护
- 可审查
- 可扩展

## 6.2 模板建议结构

建议采用 YAML 或 JSON 作为标准化描述格式。

示例：

```yaml
caseId: TC_LOGIN_TRADE_001
caseName: 登录并发起交易
description: 登录系统后发起交易，并校验交易记录写入数据库

beforeActions:
  - type: db_cleanup
    sql: "DELETE FROM trade_order WHERE test_flag = 'Y'"
  - type: db_init
    sql: "INSERT INTO user_account (...) VALUES (...)"

mainSteps:
  - type: web_open
    url: "https://test.example.com/login"
  - type: web_input
    target: "用户名输入框"
    value: "${username}"
  - type: web_input
    target: "密码输入框"
    value: "${password}"
  - type: web_click
    target: "登录按钮"
  - type: web_assert_text
    target: "首页"
    expected: "欢迎登录"
  - type: api_call
    method: POST
    url: "/api/trade/create"
    body:
      amount: 100
      currency: CNY
  - type: api_assert
    field: "code"
    expected: 0
  - type: db_assert
    sql: "SELECT count(*) as cnt FROM trade_order WHERE amount = 100 AND test_flag = 'Y'"
    expected:
      cnt: 1

afterActions:
  - type: db_cleanup
    sql: "DELETE FROM trade_order WHERE test_flag = 'Y'"
```

# 7. 核心执行链路设计

# 7.1 执行主流程

```text
接收输入
  -> 解析输入
  -> 生成结构化测试任务
  -> 生成/加载标准化脚本模板
  -> 构建执行计划
  -> 执行 beforeActions
  -> 执行 mainSteps
  -> 执行 assertions
  -> 执行 afterActions
  -> 汇总结果
  -> 生成报告
```

---

## 7.2 时序说明

### 7.2.1 登录场景执行时序
```text
用户输入测试目标
  -> AI解析登录需求
  -> 生成登录测试步骤
  -> 打开登录页
  -> 输入用户名密码
  -> 点击登录
  -> 页面断言跳转成功
  -> 接口断言登录成功
  -> 数据库校验登录相关记录（如适用）
  -> 输出报告
```

### 7.2.2 发起交易场景执行时序
```text
用户输入发起交易需求
  -> AI生成测试步骤
  -> 执行前置数据初始化
  -> 登录系统
  -> 进入交易页
  -> 填写交易信息
  -> 提交交易
  -> 页面断言提交成功
  -> 接口断言交易创建成功
  -> 数据库校验交易记录新增
  -> 执行测试后数据清理
  -> 输出测试报告
```

# 11. 异常处理设计

## 11.1 页面异常
例如：

- 页面未加载完成
- 元素未找到
- 点击无响应

处理策略：

- 显式等待
- 自动重试
- 截图保留现场
- 输出定位信息

## 11.2 API 异常
例如：

- 请求超时
- 状态码异常
- 返回字段缺失

处理策略：

- 记录请求与响应摘要
- 标记步骤失败
- 中断或按策略继续

## 11.3 数据库异常
例如：

- SQL 执行失败
- 查询结果为空
- 初始化或清理失败

处理策略：

- 记录 SQL 摘要
- 输出错误原因
- 标记执行失败
- 执行必要清理补偿

---

# 12. 安全设计

## 12.1 数据库安全
- 限制仅连接测试环境
- 限制高风险 SQL
- 敏感信息脱敏
- 记录执行行为

## 12.2 配置安全
- 数据库账号密码不明文写入代码
- 配置文件支持环境变量注入
- 日志不输出完整密钥、密码、token

## 12.3 执行安全
- 禁止误执行生产地址
- 增加环境检查机制
- 增加数据库白名单校验
